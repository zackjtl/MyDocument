核心設計理念

主流程保證正確性（透過同時只能被一個port存取的 Bitmap判斷）， Self-Audit 保證可觀測性（透過 Log 與 Summarize）。
兩者獨立運作，即使主防線失效，稽核機制仍能留下證據供人員判讀。

Critical Section 互斥機制

為了避免多台機器「同時」讀寫同一個序號範圍，系統設計了兩道鎖來確保 Critical Section（關鍵程式區段）同時間只會被一個 port 執行。

1
本機內部 Mutex

同一台電腦上若有多個 port 同時測試，先透過 Mutex 讓這些執行緒排隊。只有拿到鎖的執行緒才能繼續下一步。

2
跨機台檔案鎖（LockFile）

開啟對應的 .bin 後，透過 Windows LockFile 將整個檔案鎖定。其他機器若試圖搶鎖，會進入重試（最長約 5 秒），直到鎖被釋放或逾時失敗。

關鍵原則：「讀取 → 檢查 → 標記 → 寫入磁碟」這整串動作都在鎖定區間內完成，中間不會被其他 port 打斷。

測試流程中的兩次檢查

第一次 初始化後（InitCard 之後）

Check=true Record=false
先檢查序號是否已被標記。若重複，立即拋錯並停止測試。

中間進行各項 QC 測試（CopyCompare、SpeedClass、MKB...）
第二次 全部測試通過後

Check=true Record=true
再次確認無重複，並將該序號的 bit 標記為已使用，立即寫入磁碟。

為什麼需要檢查兩次？

第一次防止「已測過的卡片被重測」；第二次確保「測試通過後才正式佔用序號」。 若第一次就標記，中途測試失敗會導致該序號被浪費；若只檢查一次，可能讓其他機台在測試過程中搶先標記同一序號。

MECHANISM 02

SN Self-Audit 機制

i

Self-Audit 不會讓測試失敗。它是「監視器回放」的概念—— 主流程靠鎖檔案來防止序號衝突，Self-Audit 則把「誰在什麼時候進出序號操作」都記錄下來， 萬一將來發現有重複序號流出，可以回頭看是哪個環節的互斥機制失效。

📝

寫入 ENTER / EXIT

每台機器在進行序號操作前寫 ENTER，完成後寫 EXIT。兩筆記錄之間會夾住實際的 .bin 鎖檔區間。

🎫

Token 掛牌機制

開跑前在共用資料夾掛上「號碼牌」（.tok 檔），讓大家知道這一輪還有多少台機器在線。這是為了確認其他機器是否離場，自己是不是最後一台。

🔍

最後一台負責分析

當最後一台機器收工撤牌時，自動分析整份 log，產出統計摘要並備份原始檔案。

Log 分析示意圖

正常情況（同一序號依序進出）
Station-A ENTER SN=1234 → 操作 .bin → EXIT res=0 | Station-B ENTER SN=1234 → 操作 .bin → EXIT res=0
異常情況（Self-Audit 會抓到）
Station-A ENTER SN=1234 → 還沒 EXIT → Station-B ENTER SN=1234 ⚠️

Station-B 在 Station-A 尚未 EXIT 前就 ENTER → 被記為「Suspicious overlap」，表示互斥機制可能失效。

關注重點

Q1

如果 Self-Audit 發現異常，測試會不會被擋下？

不會。Self-Audit 的分析結果只會寫入 history log 與測試程式的內部記錄，不會改變該卡片的測試結果。 它的目的是「事後追查」，不是「即時攔截」。主流程的即時攔截由 .bin + LockFile 負責。

Q2

為什麼要額外設計 Self-Audit 機制

主流程（.bin + LockFile）是「防線」；Self-Audit 是「監視器」。
任何檔案鎖機制都可能因網路延遲、作業系統排程或共用資料夾異常而失效。 Self-Audit 獨立運作，即使主防線意外失效，也能留下證據供工程師還原現場。

Q3

如何人工查看 Self-Audit 結果？

在每個 Lot 的資料夾下：
SN_record_audit_history.log：每輪的統計摘要（持續累積）
backup\SN_record_audit_*.log：原始稽核 log 的備份（含時間戳與機台碼）
